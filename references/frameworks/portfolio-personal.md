# Portfolio Personal — Metric-Driven Individual Portfolio Framework

## Purpose & When To Use This Framework

Use this framework when an individual practitioner needs a print-ready or PDF-distributed portfolio that proves their work via case study, not gallery. The audience is a hiring manager, recruiter, prospective client, or collaborator who will skim the cover, read one hero case study deeply, then scan the supporting grid for breadth signals.

Per the framework-structures-2026 research backbone (§3 Portfolio framework), the portfolio must evolve from a gallery into a metric-validated narrative using "Problem-Process-Outcome" scaffolding. Hero projects get deep case studies; supporting work gets a project matrix or grid. Bio content stays secondary to the work's results.

Default page count is 12 (cover + bio + 4-page hero case study + 4-page supporting grid + skills + contact). Compress to 8 when a single hero project is exceptional. Expand to 16 when multiple hero projects each warrant a deep case study.

## Narrative Arc / Page Sequence

1. **Page 1 — Cover.** Name + discipline + one-line positioning. Example: "Frank Chimero — Designer, Writer, Educator." Visual: one portfolio thumbnail or signature work as background. No "Portfolio 2026" title — the cover IS the portfolio.
2. **Page 2 — About Me / Bio (one page).** Three to five sentences: who, what, what you care about, what you have shipped. Photo (optional, professional, not corporate-headshot stock). Bio is secondary — it earns one page, not three.
3. **Pages 3–7 — Hero Project Case Study.** One project, deep. Structure: **Problem context** (1 page — what was the brief, who was the client, what was at stake) → **Process** (1–2 pages — iterations, decisions made, alternates rejected, why) → **Outcome** (1–2 pages — final artifact, measured results, client endorsement). Outcome metrics use exact figures per the research: "47% increase in form submissions" or "23% improvement in organic traffic" — not "significant improvement."
4. **Pages 8–11 — Supporting Project Grid.** Two to four pages of project matrix. Per page: 2–4 supporting projects with thumbnail + 1-line context + 1 outcome metric. This is the breadth proof. Each project earns ~6 lines of treatment, not 6 pages.
5. **Page 12 — Skills / Tools.** Tagged list grouped by category (design tools, programming languages, soft skills with evidence). No skill bar graphs with arbitrary percentages.
6. **Page 13 — Contact CTA.** Named contact (email, LinkedIn, website, optional phone). One CTA verb: "Available for hire" / "Open to projects Q3 2026 onward" / "Let's talk — [calendar link]." QR code to portfolio website or scheduling tool.

## 7-Step Content Checklist

1. **Hero project selection — choose the project closest to the work you want next.** Portfolio is forward-looking, not retrospective. If you want to do brand systems work, your hero project must be a brand system, even if your highest-paid work was a logo. Per the research §3 hero-vs-supporting principle.
2. **Problem context names the client and the stakes.** "I designed a website" is dead. "Re-brand for a 12-year-old SaaS company entering enterprise market; brand had been DIY since founding; stakes: closing the next $5M deal in Q2." Specific client, specific stake.
3. **Process pages show alternates rejected.** A portfolio that shows only the final artifact reads as "lucky guess." Show 2–3 alternate directions explored, then explain why the final direction won. This is the "Problem-Process-Outcome" middle that most portfolios skip.
4. **Outcome uses exact figures or named endorsements.** Either measured numbers ("47% form submission lift") or a verbatim quote from the named client. "Significant improvement" / "client was happy" is a validate-fail.
5. **Supporting grid earns one outcome metric per project.** Even at thumbnail scale, each project gets a number or a named result. No "various branding projects, 2018–2023." Each entry: client name + project type + one measurable result.
6. **Skills page is honest, not aspirational.** List tools you have shipped production work with, not tools you have watched tutorials on. Group by category, no percentages. Soft skills must be evidenced ("Led 3-person team on Project X" not "Leadership").
7. **Contact CTA names availability state.** "Available for hire" / "Open to selective projects" / "Booked through Q4 2026, taking inquiries for Q1." Status signals confidence and respect for the reader's time.

## Do-NOT Patterns (3+)

1. **Gallery without context.** A grid of 20 thumbnails with no problem-process-outcome scaffolding reads as "I have made many things" and conveys nothing about fitness for the next hire. Per research §3, this is the dominant portfolio failure.
2. **Skill bar graphs with arbitrary percentages.** "Photoshop 92% / Illustrator 87% / Figma 95%" is meaningless and reads as juniorish. Replace with tagged tool list and named projects where each tool was used in anger.
3. **"Various branding work" / "Multiple clients" / "Selected projects 2018–2023" as catch-all.** Every project deserves a name and a stake. If a project cannot earn one line of named context, drop it from the supporting grid.
4. **Bio that takes 2+ pages.** Bio is secondary to work. One page maximum. Three to five sentences. Photo optional.
5. **Stock testimonial that is unattributed.** "Frank's work transformed our business" with no attribution is wallpaper. Either verbatim with name + role + company, or drop.
6. **Cover page titled "Portfolio 2026" / "My Work" / "Welcome."** Self-explanatory titles waste the cover. The cover IS the portfolio — use the name + discipline + signature visual instead.

## Anti-Slop Hard Rules

1. **Every hero project has measured outcome OR verbatim attributed quote.** Hard-fail if hero case study ends with "client was happy" or "project was a success" without a number or attribution.
2. **Visual ratio ≥ 70% per page (especially case study pages).** A portfolio that does not show the work is not a portfolio. Hero project pages should be 80%+ visual.
3. **Cover bears name + discipline, not "Portfolio."** Hard rule.
4. **Bio capped at one page.** Hard cap. If bio overflows, the operator is hiding the work behind the person.
5. **Supporting grid project entries each earn one named result.** No "various" / "selected" catch-all entries.
6. **No banned vocabulary.** Global ban list applies. Additional portfolio-personal bans: "passionate", "creative problem-solver", "thinking outside the box", "results-driven", "team player" — replace with specific evidence (named project, named outcome, named team).
